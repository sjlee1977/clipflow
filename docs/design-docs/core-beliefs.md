# 핵심 신념 — 아키텍처 결정 원칙

## 1. 에이전트 가독성 우선
코드베이스는 사람뿐 아니라 AI 에이전트가 읽고 수정한다.
모든 결정, 규칙, 패턴은 코드 또는 마크다운으로 리포지터리에 존재해야 한다.
Slack 메시지, Google Docs, 머릿속 지식은 에이전트에게 존재하지 않는 것과 같다.

## 2. wiki = 프롬프트 지식의 기록 시스템
AI 생성 품질을 높이는 모든 지식은 `wiki/`에 파일로 관리한다.
코드에 하드코딩된 프롬프트는 에이전트가 수정하기 어렵다.
파일로 분리된 프롬프트는 버전 관리되고 점진적으로 개선된다.

## 3. 폴백 패턴 (wiki → 하드코딩)
wiki 파일이 있으면 → wiki 사용
wiki 파일이 없으면 → 코드 내 하드코딩 폴백 사용
이 패턴은 기존 기능을 깨지 않고 점진적 마이그레이션을 가능하게 한다.

## 4. 단방향 의존성
pages → API → lib → external
이 방향을 역행하는 import는 금지.

## 5. 경계에서만 검증
입력 검증은 API 라우트 진입점에서만.
내부 함수는 이미 검증된 데이터를 받는다고 가정.

## 6. 지루한 기술 선택
에이전트가 모델링하기 쉬운 기술을 선택한다.
- 광범위한 학습 데이터에 있는 라이브러리 선호
- 불투명한 추상화보다 명시적 코드 선호
- 작은 유틸리티를 직접 구현하는 것이 불투명한 패키지보다 나을 수 있음
