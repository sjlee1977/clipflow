## 1. 전체 파이프라인 흐름

```
[사용자 입력]
    주제(topic) + 카테고리 + 모델 + 최소 길이
         │
         ▼
[Wiki 로딩]
    wiki/script/{category}/identity.md   → 채널 정체성
    wiki/script/{category}/tone.md       → 말투/문체
    wiki/script/{category}/principles.md → 누적 원칙
    wiki/script/{category}/stage-notes.md→ 단계 커스터마이징
    wiki/script/_shared/7-stage-structure.md → 공유 구조
    wiki/script/feedback/{category}/최신.md  → 최근 피드백
         │
         ▼
[Agent 1: 감독]  temperature 0.3
    주제 분석 → 핵심 각도 선정 → 7단계 설계
    모든 결정에 "왜(Why)" 포함 (CoT 핸드오프)
    출력: { coreAngle, whyCoreAngle, hookStrategy, whyHook, stageplan[], targetEmotion, keyData[] }
         │
         ▼
[Agent 2: 작가]  temperature 0.85
    감독의 전략 + CoT 이유를 받아 초고 작성
    카테고리별 Wiki 원칙 적용
    최소 {minLength}자 이상
         │
         ▼
[감독 ↔ 작가 토론]  감독 0.2 / 작가 0.7
    감독이 초고에서 2~3개 약점 지적 (이유 포함)
    작가가 해당 부분만 수정 (전체 재작성 금지)
         │
         ▼
[Agent 3: 프로듀서]  temperature 0.2
    훅 / 전환 / 클로징 / 길이 점검
    약한 부분만 외과적 수정
         │
         ▼
[Agent 4: SEO 최적화]  temperature 0.4
    제목 후보 3개 + 썸네일 텍스트 + 설명문 + 해시태그 + 검색 키워드
         │
         ▼
[최종 출력]
    script (최종 대본)
    strategy (감독 전략)
    directorFeedback (토론 피드백 목록)
    producerNotes (프로듀서 수정 내역)
    seo (SEO 패키지)
    steps (에이전트별 진행 기록)
```

## 2. 설계 패턴

### 2-1. ViMax 패턴 — 역할 분리
단일 LLM이 전략 + 작성 + 검토를 모두 하면 품질이 저하됨.
감독(전략) / 작가(실행) / 프로듀서(품질) / SEO(최적화)로 역할을 명확히 분리.

### 2-2. FilmAgent 패턴 — CoT 핸드오프
각 에이전트가 결과물만 넘기는 것이 아니라 **"왜 이렇게 결정했는가"의 이유**를 함께 전달.
작가가 감독의 의도를 정확히 이해하고 판단력 있게 구현할 수 있게 함.

```
감독 → 작가 전달 예시:
"5단계를 공포 심화로 구성하세요"  (❌ 이유 없음 — 작가가 맹목적 실행)
"5단계를 공포 심화로 구성하세요 — 이 시점에서 시청자의 긴장이 최고조이기 때문"  (✅ CoT)
```

### 2-3. YouTube Automation Agent 패턴 — SEO 분리
대본 완성 후 별도 에이전트가 SEO 메타데이터(제목/설명/태그)를 생성.
대본 작성 중 SEO를 신경 쓰면 자연스러운 대본 흐름이 깨지기 때문.

### 2-4. 1라운드 토론 제한
FilmAgent 원본 논문에서 권장하는 다중 토론을 1라운드로 제한.
**이유**: 2라운드 이상에서 감독이 이미 수정된 부분을 다시 지적하는 순환 문제 발생.
1라운드 내에서 핵심 2~3개 약점만 집중 수정.

## 3. 카테고리 시스템

### 3-1. 지원 카테고리

| 카테고리 | 채널 특성 | 주요 콘텐츠 |
|---------|---------|-----------|
| `economy` | 경제/투자 정보 채널 | 주가, 환율, 부동산, 글로벌 이슈 |
| `psychology` | 심리학 대중화 채널 | 인간 심리, 행동 패턴, 관계 |
| `horror` | 공포/미스터리 채널 | 도시전설, 미해결 사건, 초자연 |
| `health` | 건강/의학 정보 채널 | 질병, 식습관, 운동, 의학 연구 |
| `history` | 역사 이야기 채널 | 한국사, 세계사, 역사적 인물 |
| `general` | 다목적 채널 | 다양한 주제 커버 |

### 3-2. 카테고리별 Wiki 파일 구조

```
wiki/script/
├── _shared/
│   └── 7-stage-structure.md     ← 공통 7단계 구조 (기본값)
├── {category}/
│   ├── identity.md              ← 채널 정체성 (페르소나, 목표 시청자)
│   ├── tone.md                  ← 말투/문체/금지 표현
│   ├── principles.md            ← 누적 원칙 (개선될수록 쌓임)
│   └── stage-notes.md           ← 7단계 중 이 카테고리 전용 변형
└── feedback/
    └── {category}/
        └── YYYY-MM-DD.md        ← 최신 피드백 (자동 로드)
```

### 3-3. Wiki 파일 우선순위

```
1. {category}/stage-notes.md 존재 → 해당 카테고리 전용 단계 사용
2. 없으면 → _shared/7-stage-structure.md 기본 구조 사용
3. feedback/ 최신 파일 → 항상 "반드시 반영" 섹션으로 삽입
4. principles.md → 시스템 프롬프트 마지막에 추가 삽입
```

## 4. 7단계 대본 구조

**파일**: `wiki/script/_shared/7-stage-structure.md`
(경제/일반 카테고리 기본. 다른 카테고리는 stage-notes.md로 일부 재정의)

### 1단계: 도입부 — 훅 & 몰입 유도

**목표**: 인사 없이 첫 문장부터 바로 시작. 60초 안에 4단계를 속도감 있게 몰아치기.

**훅 형식 3가지** (반드시 이 중 하나로 시작):
```
형식 A: "[핵심 주제], 지금 제대로 알고 있어?"
형식 B: "[충격적 사실/수치], 이게 우리한테 무슨 의미인지 알아?"
형식 C: "지금 [시장 상황]인데, 대부분이 완전히 잘못 보고 있어."
```

**금지 사항**:
- 인사말 ("안녕하세요", "돌아왔습니다")
- 채널 소개
- 오늘 주제 예고 형식의 도입

**필수 포함**:
- 긴박감 & 희소성 부여 (2~3문장)
- 채널 시그니처 멘트 (고정)
- 본론 진입 전환

---

### 2단계: 상황 정리 — 팩트 체크

**목표**: 시청자가 맥락을 완벽히 이해하도록 배경 설정.

**데이터 의무 규칙**:
```
금지: "최근 많이 올랐다", "요즘 분위기가 안 좋다"  (모호한 표현)
필수: [날짜] + [지수/가격/퍼센트] + [변화 방향과 폭]  (구체적 수치)
```

**팩트 의미 번역 (의무)**:
숫자 제시 직후 반드시 해석 제공:
```
"이게 무슨 의미냐면..."  또는  "이게 우리한테 어떤 뜻이냐면..."
```

---

### 3단계: 쉬운 설명 — 개념 번역 구간

**목표**: 전문 용어와 복잡한 구조를 누구나 이해하는 일상 언어로 번역.
**기준**: "초등학생이 들어도 고개를 끄덕일 수 있는가?"

**활용 가능한 비유 소재**:
음식(피자, 치킨), 가게 운영, 집 구매, 마트 쇼핑, 학교생활, 스포츠

**어려운 용어 첫 등장 시 의무 규칙**:
```
영어 약어 첫 등장 → 반드시 한국어 풀이를 괄호 안에 병기
예: CPI(소비자물가지수), ETF(상장지수펀드)
```

---

### 4단계: 고정관념 타파 — 핵심 인사이트

**목표**: 영상 전체의 클라이맥스. 대중의 착각을 저격하는 구간.

**패턴**:
```
대중의 착각 저격 → 낡은 공식 vs 현재 구조 대비 → 찰떡 비유로 마무리
"여기서 진짜 많은 분들이 걸려드는 함정이 있어."
"한마디로 정리하면, [비유]이랑 똑같은 거야."
```

---

### 5단계: 실전 시나리오 & 체크리스트

**금지**: "무조건 오른다/내린다" 단정 예측
**의무**: 조건부 If-Then 구조 (시청자가 스스로 판단하게 함)

**3갈래 시나리오 트리**:
```
낙관: [조건 A]가 충족되면 → [목표 수치/방향] 가능
중립: [조건 B] 상황 유지되면 → [횡보/박스권]
비관: [최악의 변수 C]가 터지면 → [하단 지지선/방어 구간]
```

---

### 6단계: 리스크 점검 — 브레이크 구간

**이 구간은 선택이 아님. 모든 영상에 반드시 포함.**

**고정 전환 멘트**:
```
"자, 여기서 끝내면 안 돼. 좋은 것만 보는 건 투자가 아니야. 리스크도 정직하게 봐야지."
```

**필수 포함**:
- 리스크 넘버링 (최소 2개 이상, 각각 수치 + 영향 포함)
- 최악 시나리오 묘사
- 대응 안전망 제공 ("탈출 신호는 [지표]가 [기준값]을 [조건]으로 확인될 때야")

---

### 7단계: 마무리 — CTA & 클로징

**필수 포함**:
1. 투자 철학 한 줄 요약 (여운)
2. 좋아요 & 댓글 유도 (당당하게)
3. 제작 노력 어필
4. 시그니처 클로징 (채널별 고정 — 절대 변경 금지)

## 5. Agent 1: 감독

**역할**: 주제 분석 → 핵심 각도 선정 → 7단계 전략 설계
**Temperature**: 0.3 (전략적 일관성 우선)
**max_tokens**: 2500

**페르소나**:
100편 이상의 유튜브 영상을 기획한 콘텐츠 감독.
주제를 보면 어떤 각도로 가야 바이럴이 되는지 즉시 안다.
이유 없는 지시는 작가가 판단할 수 없으므로 모든 결정에 "왜(Why)"를 포함.

**시스템 프롬프트**:
```
당신은 유튜브 콘텐츠 감독입니다. 100편 이상의 영상을 기획했습니다.
주제를 받으면 어떤 각도로 가야 바이럴이 되는지 즉시 압니다.
모든 결정에는 반드시 "왜(Why)"를 포함합니다. 이유 없는 지시는 작가가 판단할 수 없습니다.

채널 정체성:
{wiki/identity.md}

7단계 구조 참고:
{wiki/_shared/7-stage-structure.md}

카테고리 특화 지침:
{wiki/stage-notes.md}

JSON만 출력. 다른 텍스트 없음.
```

**사용자 프롬프트**:
```
주제: "{topic}"
카테고리: {category}
목표 길이: 최소 {minLength}자

이 주제로 최고의 유튜브 대본을 만들기 위한 전략을 설계하세요.

{
  "coreAngle":     "이 주제를 어떤 각도로 접근할지 (1~2문장)",
  "whyCoreAngle":  "왜 이 각도인가 — 시청자 심리와 경쟁 채널 대비 이유",
  "hookStrategy":  "첫 15초 훅 전략 (구체적 문장 or 장면 묘사)",
  "whyHook":       "왜 이 훅인가",
  "stageplan": [
    { "stage": 1, "title": "단계명", "content": "이 단계에서 다룰 내용", "why": "왜 이 내용인가" },
    ...7단계 전부...
  ],
  "targetEmotion": "시청자가 영상 끝에 느껴야 할 감정",
  "keyData":       ["반드시 포함할 데이터/사례 1", "2", "3"]
}
```

## 6. Agent 2: 작가

**역할**: 감독의 전략 + CoT 이유를 받아 실제 대본 초고 작성
**Temperature**: 0.85 (창의적 글쓰기 — 다양성 최대)
**max_tokens**: 5000

**페르소나**:
카테고리별 전문 대본 작가.
감독의 전략을 완벽히 이해하고 의도를 살려 실행.

**시스템 프롬프트**:
```
당신은 유튜브 대본 전문 작가입니다.
감독의 전략과 이유를 완벽히 이해하고, 그 의도를 살려 실제 대본으로 구현합니다.

채널 정체성:
{wiki/identity.md}

톤 & 말투:
{wiki/tone.md}

축적된 원칙:
{wiki/principles.md}

최근 피드백 (반드시 반영):
{wiki/feedback/최신.md}

절대 규칙:
- 마크다운 없이 순수 텍스트만 (제목, 굵게, 기호 금지)
- 단락 구분은 줄바꿈만
- 존댓말/방송 앵커 언어 금지
- 최소 {minLength}자 이상 작성
- 모든 어려운 개념은 일상 비유로 치환
- 숫자 없는 주장 금지 (%, 날짜, 수치 필수)

대본만 출력. 단계 제목, 메모, 설명 없음.
```

**사용자 프롬프트** (감독 CoT 전달 구조):
```
주제: "{topic}"

감독의 전략:
핵심 각도: {coreAngle}
이유: {whyCoreAngle}

훅 전략: {hookStrategy}
이유: {whyHook}

단계별 설계:
1단계 [단계명]: {content}
  → 감독 지시 이유: {why}
2단계 [단계명]: ...
(7단계까지 전부)

목표 감정: {targetEmotion}
반드시 포함할 데이터: {keyData}

위 감독 지시를 완벽히 따라 대본을 작성하세요. 최소 {minLength}자.
```

**작가 절대 규칙 상세**:

| 규칙 | 내용 |
|------|------|
| 마크다운 금지 | 제목(#), 볼드(**), 목록(-) 등 모든 마크다운 기호 사용 금지 |
| 순수 텍스트 | 단락 구분은 줄바꿈만 — 영상 자막/TTS에 그대로 사용되기 때문 |
| 존댓말/앵커체 금지 | "~습니다", "~입니다" 형식 금지, 친근한 반말(해라체) 사용 |
| 숫자 의무 | "많다", "높다" 등 모호한 표현 금지, 반드시 구체적 수치 포함 |
| 비유 의무 | 전문 용어 첫 등장 시 일상 비유 또는 괄호 풀이 필수 |

## 7. 감독 ↔ 작가 토론

**패턴**: FilmAgent 1-round Discussion
**라운드**: 1회 고정 (2라운드 이상 시 순환 지적 발생)

### 7-1. 감독 검토 단계

**Temperature**: 0.2 (정확한 문제 파악)
**max_tokens**: 1000

```
시스템: "당신은 엄격한 유튜브 콘텐츠 감독입니다. 초고를 검토하고 
         정확히 2~3개의 개선 지시만 내립니다. 모든 지시에는 반드시 이유를 포함합니다."

입력: 원래 전략 + 초고 (앞 3000자)

출력:
{
  "feedback": [
    {
      "issue":       "문제점",
      "location":    "어느 부분",
      "instruction": "구체적 수정 지시",
      "why":         "왜 수정해야 하는가"
    }
  ]
}
```

**검토 기준**:
- 핵심 각도(coreAngle)가 대본에 실제로 구현되었는가?
- 목표 감정(targetEmotion)이 클로징에서 달성되었는가?
- 각 단계의 "왜" 의도가 대본에 살아있는가?

### 7-2. 작가 수정 단계

**Temperature**: 0.7 (수정의 창의성 허용)
**max_tokens**: 5000

```
시스템: "당신은 대본 작가입니다. 감독의 피드백을 받아 지적된 부분만 수정합니다.
         전체 재작성 금지. 대본 전문만 출력."

입력: 감독 피드백 (번호 + 위치 + 지시 + 이유) + 원본 대본
```

## 8. Agent 3: 프로듀서

**역할**: 최종 품질 점검 + 약한 부분만 외과적 수정
**Temperature**: 0.2 (정확한 수정, 전체 재작성 방지)
**max_tokens**: 6000

**페르소나**:
최종 품질 책임자. 시청자 이탈 구간을 본능적으로 안다.
전체를 새로 쓰는 것이 아니라 약한 부분만 수술적으로 수정.

**점검 4항목**:

| 항목 | 판단 기준 |
|------|---------|
| 훅 강도 | 첫 15초가 시청자를 붙잡는가? |
| 전환 자연스러움 | 각 단계 사이 흐름이 매끄러운가? |
| 클로징 연결 | 구독 CTA가 자연스럽게 연결되는가? |
| 길이 충족 | 요청한 최소 길이({minLength}자) 이상인가? |

**시스템 프롬프트**:
```
당신은 유튜브 콘텐츠 프로듀서입니다. 시청자 이탈 구간을 본능적으로 압니다.
대본의 훅/전환/클로징을 점검하고 약한 부분만 수정합니다. 전체 재작성 금지.
JSON만 출력.
```

**출력 형식**:
```json
{
  "producerNotes": ["수정 항목1", "수정 항목2"],
  "finalScript": "수정된 최종 대본 전체"
}
```

## 9. Agent 4: SEO 최적화

**역할**: 완성된 대본 기반으로 YouTube SEO 메타데이터 패키지 생성
**Temperature**: 0.4 (SEO와 창의성의 균형)
**max_tokens**: 800

**페르소나**:
YouTube SEO 전문가. CTR(클릭률)과 검색 노출을 동시에 최적화.
대본이 완성된 후 분리 실행 — 대본 작성 중 SEO를 신경 쓰면 자연스러운 흐름이 깨지기 때문.

**시스템 프롬프트**:
```
당신은 YouTube SEO 전문가입니다. CTR(클릭률)과 검색 노출을 동시에 최적화합니다.
대본을 분석해서 SEO 패키지를 생성합니다. JSON만 출력.
```

**사용자 프롬프트**:
```
주제: {topic}
카테고리: {category}
대본 첫 500자: {script.slice(0, 500)}
```

**출력 형식 및 기준**:

```json
{
  "titles": [
    { "title": "제목안 1 (호기심 유발형)", "reason": "이 제목이 클릭되는 이유" },
    { "title": "제목안 2 (숫자/데이터형)", "reason": "..." },
    { "title": "제목안 3 (반전/충격형)",   "reason": "..." }
  ],
  "thumbnailText":   "썸네일 텍스트 (5~8자, 강렬하게)",
  "description":     "영상 설명문 첫 3줄 (검색 노출용, 핵심 키워드 포함)",
  "hashtags":        ["#태그1", ... "#태그10"],
  "searchKeywords":  ["검색 키워드1", ... "키워드5"]
}
```

**제목 3가지 유형**:

| 유형 | 원리 | 예시 패턴 |
|------|------|---------|
| 호기심 유발형 | 답을 알고 싶게 만드는 정보 격차 | "왜 ~인지 99%가 모른다" |
| 숫자/데이터형 | 구체성으로 신뢰감 + CTR 향상 | "3가지 이유로 ~이 무너진다" |
| 반전/충격형 | 기존 상식과 반대되는 주장 | "~해야 한다고? 오히려 역효과" |

**썸네일 텍스트 기준**: 5~8자, 최대한 강렬하게. 시청자가 썸네일에서 즉각 반응해야 함.

**해시태그 전략**: 10개, 대주제 2~3개 + 소주제 4~5개 + 롱테일 2~3개 혼합

## 10. Wiki 로딩 시스템

### 10-1. 카테고리별 로딩 순서

```typescript
function loadCategoryWiki(category: string) {
  return {
    identity:   readWiki(`${category}/identity.md`),    // 채널 정체성
    tone:       readWiki(`${category}/tone.md`),         // 말투/문체
    structure:  readWiki('_shared/7-stage-structure.md'),// 공통 구조
    stageNotes: readWiki(`${category}/stage-notes.md`), // 단계 커스터마이징
    principles: readWiki(`${category}/principles.md`),  // 누적 원칙
    feedback:   getLatestFeedback(category),             // 최신 피드백
  };
}
```

### 10-2. 피드백 자동 로딩

`wiki/script/feedback/{category}/` 폴더에서 파일명 기준 **가장 최신 파일 1개** 자동 로드.
파일명 형식: `YYYY-MM-DD.md` (날짜 내림차순 정렬 후 첫 번째)

### 10-3. Fallback 처리

파일이 없으면 → 코드 내 하드코딩 기본값 사용:
```
identity 없음   → 카테고리명으로 대체
tone 없음       → "친근한 반말 (해라체)"
structure 없음  → "도입훅 → 팩트체크 → 개념번역 → 인사이트 → 시나리오 → 리스크 → 클로징"
stageNotes 없음 → 기본 구조 사용
principles 없음 → 시스템 프롬프트에서 제외
feedback 없음   → 최근 피드백 섹션 제외
```

### 10-4. Principles 파일 운영 원칙
- 누적형: 새 원칙 발견 시 추가, 삭제하지 않음
- 시스템 프롬프트 마지막에 추가로 삽입 (다른 지침보다 낮은 우선순위)
- "비워둠" 텍스트가 있으면 삽입 스킵

| 에이전트 | Temperature | 이유 |
|---------|------------|------|
| 감독 | **0.3** | 전략적 일관성 필요 — 창의적 각도 설계지만 논리성 우선 |
| 작가 | **0.85** | 창의적 글쓰기 — 다양하고 생생한 표현 최대화 |
| 감독(토론 검토) | **0.2** | 정확한 약점 파악 — 창의성보다 비판적 판단 |
| 작가(토론 수정) | **0.7** | 수정의 창의성 허용 — 너무 낮으면 기계적 수정 |
| 프로듀서 | **0.2** | 외과적 수정 — 전체 재작성 방지 |
| SEO | **0.4** | SEO 원칙 + 약한 창의성 균형 |

## 12. 설계 판단 기록

### 감독과 작가를 분리한 이유
단일 LLM이 전략 설계와 대본 작성을 동시에 하면:
1. 전략을 세우다가 글 쓰는 쪽으로 이탈 (전략이 얕아짐)
2. 대본 쓰다가 초반 전략을 잊어버림 (구조가 무너짐)

감독이 전략을 고정하고 작가가 실행하면 두 가지 모두 품질 향상.

### CoT(Chain of Thought) 핸드오프를 넣은 이유
"무엇을" 만이 아니라 "왜"를 전달하면 작가 에이전트가 맹목적 실행이 아닌
판단력 있는 실행을 함. 세부 상황에서 자율적 결정이 가능해짐.

### 1라운드 토론 제한 이유
다중 라운드 시 발생 문제:
- 감독이 이미 수정된 부분을 다시 지적하는 순환
- 후반 라운드로 갈수록 지적이 작아지고 의미 없어짐
- 작가가 점점 보수적으로 변해 창의성 하락

1라운드에 핵심 2~3개만 집중 수정이 품질/비용 최적.

### SEO를 마지막에 분리한 이유
대본 작성 중 SEO를 신경 쓰면:
- 제목에 맞추느라 자연스러운 대본 흐름이 깨짐
- 키워드 삽입이 강제되어 대사가 부자연스러워짐

완성된 대본을 보고 SEO를 역산하는 방식이 품질 높음.

### 프로듀서가 전체 재작성을 금지하는 이유
감독+작가+토론을 통해 완성된 구조와 문체를 프로듀서가 전부 갈아엎으면
앞의 모든 에이전트 작업이 무의미해짐.
"약한 부분만 수술적으로 수정"이 핵심 원칙.

### 피드백 파일을 자동 로딩하는 이유
운영하면서 발견한 개선 사항을 코드 수정 없이 wiki 파일 업데이트만으로 반영.
"반드시 반영" 섹션으로 삽입되어 가장 높은 우선순위.
