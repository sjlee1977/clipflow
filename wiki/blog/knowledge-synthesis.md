# 지식 연결 원칙 — 위키를 살아있게 만드는 법

> 이 파일은 Karpathy의 LLM Wiki 패턴에서 도출한 위키 유지·운영 원칙이다.
> 에이전트가 "어떻게 써야 하는가"가 아닌 "위키를 어떻게 관리해야 하는가"를 다룬다.

---

## 핵심 개념: LLM이 직접 유지하는 위키

### 왜 RAG 대신 위키인가

RAG(검색 증강)는 원본 문서에서 관련 청크를 찾아 붙여준다.
LLM 위키는 LLM이 원본을 소화해 자신의 언어로 재작성한 지식베이스다.

| | RAG | LLM Wiki |
|--|-----|----------|
| 지식 형태 | 원본 청크 그대로 | LLM이 합성한 마크다운 |
| 업데이트 | 원본 문서 변경 시 자동 | 운영자(LLM)가 직접 편집 |
| 연결성 | 문서 간 관계 없음 | 교차 참조로 연결 |
| 신뢰 위험 | 검색 실패 | LLM이 만든 요약에 의존 (주의) |

**핵심 원칙**: 이 위키에 적힌 것을 LLM이 쿼리할 때, 원본을 직접 읽는 것보다 오류 가능성이 낮아야 한다. 위키가 원본보다 명확하고 구조적이어야 한다.

## 프론트매터 스키마 — 모든 wiki 파일의 메타데이터 규격

모든 wiki 파일은 이 YAML 프론트매터로 시작한다. LLM과 유지 관리자 모두가 읽는 기계·인간 공용 메타데이터.

```yaml
---
updated: YYYY-MM-DD          # 마지막 실질적 수정일
ttl: 90                      # 유통기한 (일). 이 기간 후엔 검토 필요 상태
ttl_reason: 한 줄 이유       # 왜 이 TTL인지 — 판단 근거
volatile: false              # true = 외부 의존(SEO 알고리즘 등)으로 예고 없이 만료 가능
conflicts_with:              # (선택) 다른 파일과 충돌하는 필드가 있을 때
  - file: 다른파일.md
    field: 충돌하는 항목명
    authority: 권위파일.md   # 어느 쪽이 기준인지
    resolution: 해소 방법
split_from: 원본파일.md      # (선택) 분할로 생긴 파일일 때 원본 명시
split_children:              # (선택) 분할된 자식 파일이 있을 때
  - file: 자식파일.md
    contains: 이동된 섹션 목록
---
```

### TTL 가이드라인

| 유형 | TTL | 예시 |
|------|-----|------|
| 불변 원칙 (독자 심리 기반) | 365일 | hook-writing, structure, emotional-flow |
| 반안정 원칙 (정책/AI 모델 영향) | 180일 | writer-persona, cta-writing |
| SEO·알고리즘 의존 | 90일 | seo-principles |
| 코드 동기화 필요 | 60일 | evaluation-rubric, multi-agent-roles, pipeline guides |
| 목차·인덱스 | 30일 | wiki/index.md |

### TTL 만료 시 처리

만료 ≠ 즉시 삭제. 검토 후:
1. 내용이 여전히 유효하면 → `updated` 날짜만 갱신
2. 내용이 변경됐으면 → 업데이트 후 `updated` 갱신
3. 내용이 폐기됐으면 → 파일 삭제 + index.md에서 제거

### 분할 기준 (split_threshold: 400줄)

400줄을 초과하면 분할을 검토한다.
- 400줄 = LLM이 단일 컨텍스트로 편하게 처리할 수 있는 상한
- 분할 시 원본에 `split_children`, 자식에 `split_from` 필드 추가
- 분할 후 양쪽 파일 모두 "## 관련 문서"로 연결

## 3계층 지식 구조

```
Layer 1: Raw Sources (원본)
  ← 사용자 피드백, 블로그 성과 데이터, AI 연구 논문
  ← Karpathy gist, 업계 사례, 내부 실험 결과

Layer 2: Wiki (합성된 지식)  ← 지금 이 파일들
  ← wiki/blog/*.md
  ← LLM이 원본을 소화해 재작성한 원칙들

Layer 3: Schema (운영 규칙)
  ← CLAUDE.md, ai-blog-pipeline-guide.md
  ← AI가 실제로 작동하는 규칙 (프롬프트, 파라미터)
```

**방향은 단방향**: Raw Sources → Wiki → Schema
Wiki가 Schema에 삽입되고, Schema가 에이전트를 구동한다.

## 지식 연결의 3가지 원칙

### 원칙 1: 모든 파일은 관련 파일을 명시한다

각 wiki 파일의 마지막 섹션은 반드시 "## 관련 문서"여야 한다.
링크만 나열하지 않고, 왜 연결되는지 한 줄로 설명한다.

나쁜 예:
```
## 관련 문서
- structure.md
- hook-writing.md
```

좋은 예:
```
## 관련 문서
- `structure.md` — 1막(훅&세계설정)에서 훅이 사용되는 맥락
- `evaluation-rubric.md` — 1번 훅 강도 차원 채점 기준 (9~10점 조건)
```

### 원칙 2: 변경은 연쇄적으로 확인한다

한 파일을 변경할 때, 그 파일을 참조하는 다른 파일도 함께 확인한다.

예: `evaluation-rubric.md`의 차원을 변경하면:
- `multi-agent-roles.md`의 Agent 4 설명 확인
- `ai-blog-pipeline-guide.md`의 실제 프롬프트 확인
- 이 index.md의 Lint 체크리스트 확인

### 원칙 3: 모순은 권위 문서 기준으로 해소한다

두 파일이 충돌할 때 우선순위:
1. `ai-blog-pipeline-guide.md` — 실제 구현의 단독 권위 문서
2. `evaluation-rubric.md` — 채점 기준 권위 문서
3. 나머지 원칙 파일들 — 같은 위계, 최신 업데이트가 우선

## 지식 합성 품질 기준

위키 파일이 좋은 품질인지 판단하는 기준:

### 합성 (Synthesis) vs 나열 (Listing)

나쁜 합성 (단순 나열):
```
훅 유형:
- A형: 도발적 질문
- B형: 충격 수치
- C형: 착각 지적
```

좋은 합성 (연결과 맥락):
```
훅 유형 선택 기준:
- 정보성 글 → B형 (수치가 독자의 검색 의도를 즉시 충족)
- 공감 중심 글 → C형 (독자의 경험을 먼저 대신 말해줌)
- 설득 글 → A형 (기존 상식에 도전해 관점 전환 유도)

훅 유형은 emotional-flow.md의 1단계(긴장)와 연결됨:
A형은 인지적 긴장, B형은 수치 충격, C형은 감정적 공감으로 긴장을 만든다.
```

### 자기 충족성 (Self-Sufficiency)

각 파일은 그 파일만 읽어도 AI가 해당 작업을 수행할 수 있어야 한다.
"나머지는 다른 파일 참조"는 허용되지만, 핵심 판단 기준은 파일 내에 있어야 한다.

## 할루시네이션 위험 — 위키의 구조적 약점

LLM이 자신이 만든 요약(위키)을 쿼리할 때 주의할 점:

1. **원본보다 단순화된 정보를 사실로 확신할 위험**
   → 중요한 판단(프롬프트, 파라미터)은 항상 `ai-blog-pipeline-guide.md`를 직접 확인

2. **오래된 위키를 최신 상태로 오인할 위험**
   → 각 파일의 업데이트 날짜를 확인, 의심스러우면 실제 코드 파일 참조

3. **교차 참조 없는 파일을 독립적 사실로 오인할 위험**
   → "## 관련 문서"가 없는 파일은 불완전한 것으로 간주

**안전 원칙**: 위키는 방향을 제시하고, 코드(route.ts)가 최종 진실이다.
